> 📄 상세 API 사양 / 응답 JSON / 22개 컬럼 정의 / 코드표: `references/pbm110mw-api-spec.md`

## 🚨 사용자 확정 조회 규칙 (2026-06-03) — #3 / #4만 사용

| 케이스 | STC_CST_COD | REQ_CST_COD | REQ_DAT_TYPE | 의미 | 예시 |
|---|---|---|---|---|---|
| **#3** | `217273` | `""` (빈값) | `1` (요청일자) | 우리 회사(217273)가 KPP에 **입고 요청한** 내역 (출하) | 5/1~6/30 조회 시 0건 |
| **#4** ⭐ 기본값 | `""` (빈값) | `217273` | `2` (하차요청일자) | 우리 회사(217273)가 KPP로부터 **입고받은** 내역 (수령) | 6/1~6/30 조회 시 1건 (5/29→6/5) |
| ~~#1~~ (제거) | `217273` | `217273` | `1` (요청일자) | ❌ 모순 — 자기 자신에게 요청 | 항상 0건 |
| ~~#2~~ (제거) | `217273` | `217273` | `2` (하차요청일자) | ❌ 모순 | 항상 0건 |

**모순 이유 (사용자 설명, 2026-06-03)**: KPP의 PBM110MW는 "A 회사(STC)가 B 회사(REQ)에게 팔레트 입고 요청" 구조. STC=REQ=217273이면 "217273이 217273에게 요청" = 실제 데이터 0건이 정당. **자동화 시 무조건 #3 또는 #4만 사용.**

## 사용자 화면 캡처로 확정된 사실

### 1. UI 라벨 ↔ REQ_TYP 매핑 (사용자 화면 기준)
| UI 표시 | REQ_TYP | 의미 |
|:---:|:---:|---|
| **입고** | `01` | 납품 (코드 명칭) |
| **반납** | `02` | 반납 |

**자동화 JSON 응답의 `REQ_TYP`은 코드 값(`"01"` / `"02"`)** 그대로 반환. 사용자 보고 시 반드시 UI 라벨("입고"/"반납") 풀네임 사용 (운영원칙: 코드명만 쓰지 말 것).

### 2. 라디오 버튼 2개 (사용자가 헷갈려서 재지정함, 2026-06-03 확정)
| 라디오 | REQ_DAT_TYPE | 의미 (사용자 정의) | 정렬 키 |
|:---:|:---:|---|---|
| **요청일자** | `1` | **"내가 KPP에 요청한 날"** — 사용자가 KPP에 입고 요청 **신청한 시점** | `REQ_DAT` |
| **하차요청일자** | `2` | **"우리 회사(217273)에 실제 입고된 날"** — KPP가 우리 회사로 팔레트 입고 **완료한 시점** | `UNLOAD_REQ_DAT` |

**사용자 기본 = 하차요청일자 (REQ_DAT_TYPE=2)**. 자동화 시 기본값을 `2`로 잡는 것이 안전.

**예 (같은 1건의 두 다른 시점)**: 5/29 요청 → 6/5 입고
- 요청일자(1) 기준 5/1~5/31 → 5/29에 포함
- 하차요청일자(2) 기준 6/1~6/30 → 6/5에 포함

### 3. 자동 일자 규칙 (fn_new, line 1031~1064, 2026-06-03 확정)

```javascript
const tomorrow = new Date();
tomorrow.setDate(tomorrow.getDate() + 1);
const nowDate = gfn_today();                              // 오늘 → REQ_DAT
const tomorrowDate = ...;                                  // 내일 → UNLOAD_REQ_DAT
```

| 필드 | 값 | 규칙 |
|---|---|---|
| `REQ_DAT` (요청일자) | **오늘 (gfn_today())** | 사용자가 직접 입력 또는 자동 |
| `UNLOAD_REQ_DAT` (하차요청일자) | **자동으로 REQ_DAT의 다음 날** | "당일+1일 입고"가 KPP 기본 운영 |

**예**: 6/3 요청 → 6/4 입고 (자동). 이 규칙 모르면 6/5 건을 6/4로 옮길 때 신규 등록 시 일자 헷갈림.

### 4. ALC_STATUS (배차진행상태) 코드표 (KPP 공식, 2026-06-03 확정)

출처: `https://wpps.logisall.com/resources/js/cmm/code.js` 1052~1057행

```js
// 배차진행상태(01-요청, 02-배차요청, ...)
const GC_CDX_ALC_STATUS = "018";
const GC_CDD_ALC_STATUS_NO_ALC = "018001";    // 미배차
const GC_CDD_ALC_STATUS_TRN_REQ = "018002";   // 운송의뢰
const GC_CDD_ALC_STATUS_LOAD_WAIT = "018003"; // 상차대기
const GC_CDD_ALC_STATUS_TRN_END = "018004";   // 배차완료 (입고 종료)
```

**DB에는 마지막 2자리만 저장됨** ("018003" → "003" 또는 "03"). PBM110MW HTML의 `lockStatus = ["02","03","04"]` = "잠금 처리되는 상태" (수정 불가).

| DB 코드 | 전문 코드 | 의미 | 6/3 시점 우리 회사 데이터 |
|:---:|:---:|---|---|
| `01` | 018001 | 미배차 | — |
| `02` | 018002 | 운송의뢰 | — |
| `03` | 018003 | **상차대기** (진행 중) | 6/5 입고 예정 1건 |
| `04` | 018004 | **배차완료** (입고 종료) | 5/4, 5/11, 5/15, 5/21, 5/29 입고 완료 5건 |

**진행 순서**: 01(미배차) → 02(운송의뢰) → 03(상차대기) → 04(배차완료)

### 3. 6월 vs 5월 데이터 편중
- 6/1~6/30, **요청일자 기준**: 2,571건
- 6/1~6/30, **하차요청일자 기준**: 4,768건
- 차이 큼 — 월초 데이터(6/1~6/5)가 하차요청일자 기준에는 6월 데이터로, 요청일자 기준에는 5월 데이터로 분류됨

## 진술 vs 실제 빈도 불일치

| 진술 (사용자) | 실제 (2026-06-03 5월 조회) |
|---|---|
| "일주일에 한 번 정도" | 5/5, 5/29 = 24일 간격 (한 달에 한 번) |

→ **일주일 단위가 아닌 한 달 단위로 자동화 스케줄링 권장**

## 0건 크로스체크 패턴 (운영원칙 검증)

`STC_CST_COD=217273` + `REQ_CST_COD=217273` 직접 필터로 0건일 때, **빈 파라미터(`STC_CST_COD=""` + `REQ_CST_COD=""`) 전체 조회 후 우리 회사 0건 확인** 패턴. 사용자 운영원칙 "안다고생각하지말고항상확인검증" 충족.

빈 파라미터 응답 크기:
- 6월: 2.2~2.6MB (요청일자) / 4.7MB (하차요청일자)
- 5월: 24.6MB (요청일자)
- 응답에서 `STC_CST_COD=="217273" or REQ_CST_COD=="217273"` 필터링

## 🚨 삭제(DELETE) 차단 (2026-06-03 사용자 확인)

PBM110MW의 `fn_deleteCheck` (HTML line 1179~1198) 4가지 차단 조건:

| # | 조건 | 실패 메시지 |
|---|---|---|
| 1 | `REQ_CST_COD == ''` | "삭제 불가능한 데이터입니다" |
| 2 | `REQ_DAT == ''` | "삭제 불가능한 데이터입니다" |
| 3 | `PRD_COD == ''` | "삭제 불가능한 데이터입니다" |
| 4 | `REQ_SEQ == ''` | "삭제 불가능한 데이터입니다" |

**추가 차단 요인**:
- 백엔드 응답에서 `REQ_CST_COD=None` (화면 UI엔 "요청거래처=217273"으로 보이지만 JSON엔 누락) → **fn_deleteCheck 1번에서 차단**
- `MNG_GRD=05` 사용자 (일반 권한) → KPP 운영 정책상 진행 중 데이터 임의삭제 불가
- `lockStatus = ["02","03","04"]` → **이미 03/04 상태면 잠금** (= 진행 중이거나 완료된 건은 삭제 불가)
- ALC_STATUS=03 (상차대기)나 04 (배차완료) 모두 잠금 대상

**DELETE API**: `DELETE /ps/PBM110MW.do` + `Content-Type: application/json` + `body: JSON.stringify(deleteData)` (line 1208~1225)

**결론 (2026-06-03)**: 우리 회사(MNG_GRD=05) 권한으로는 PBM110MW 데이터 **삭제 사실상 불가**. 일자 변경 필요 시 → **신규 등록 + KPP 관리자에게 기존 건 처리 요청** 패턴.

### fn_saveCheck 필수 검증 (전부 통과해야 INSERT)
- ✓ `UNLOAD_REQ_DAT` 필수 + 일요일(`0`)이면 거부, 토요일(`6`)이면 confirm
- ✓ `UNLOAD_REQ_TIM` 필수
- ✓ `PRD_COD` 필수
- ✓ `REQ_QTY` 필수 + 0이면 거부
- ✓ `ZIP1_NUM`/`ZIP2_NUM`/`REQ_ARV_ZIP_ADDR` 필수
- ✓ `CAR_OWN_TYP=02` (USER운송)면 `RTN_DEP_CST_COD`, `RTN_ZIP1_NUM`, `RTN_ZIP2_NUM`, `RTN_ZIP_ADDR`, `DEP_CAR_USE_TIME` 추가 필수
- ✓ `CAR_OWN_TYP=01` (KPP운송)이면 위 USER운송 체크 면제

### INSERT URL/메서드
- `POST /ps/PBM110MW.do` + `Content-Type: application/json` + `body: JSON.stringify(payload)`
- 성공 시 `data.flag=true` → "저장되었습니다" → `fn_search()` 자동 재조회

## ⚠️ 데이터 변경/생성 작업 전 mandatory-verification (운영원칙)

사용자 운영원칙 (USER.md) "절대 지시 없이 데이터 생성/취소/변경 금지", "수정해 들으면 mandatory-verification Step1~5 선행, 사용자 '해'까지 대기":

| Step | 항목 | PBM110MW 신규등록 적용 |
|---|---|---|
| 1 | 사용자 지시 명확 | "신규 등록해줘" (구체적 일자/수량) |
| 2 | 데이터 변경인가? | INSERT = "데이터 생성" |
| 3 | 표준 운영 검증 | KPP fn_save 패턴 + 필드 매핑 |
| 4 | 위험도 평가 | 중간 (잘못 INSERT 시 취소 어려움, fn_delete 사실상 불가) |
| 5 | 사용자 "해" 받기 | **페이로드 먼저 보여주고 "해" 받은 후 실행** |

**운영원칙 위반 위험**: 데이터 INSERT는 거의 되돌릴 수 없음 (fn_delete 차단 + MNG_GRD=05). 페이로드 미리 보여주고 사용자 승인 후 실행이 안전.

**예외**: 사용자가 명시적으로 "이번 한 번만 페이로드 먼저 보여달라"고 하면 그 패턴 따르기 (2026-06-03 사용자 확인).
