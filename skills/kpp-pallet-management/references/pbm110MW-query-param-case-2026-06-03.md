# PBM110MW.do 조회 파라미터 — 대소문자 함정 (2026-06-03 실측)

## 🚨 핵심 발견

스킬 본문(SKILL.md)의 "기존 데이터 조회" 예제는 고객사 파라미터를 **대문자**로 표기하고 있음:

```python
# ❌ 스킬 본문 예제 (틀림)
params = {
    "REQ_DAT_FR": "20260601",
    "REQ_DAT_TO": "20260630",
    "STC_CST_COD": "217273",   # ← 이걸로 보내면 0건 응답
    "REQ_CST_COD": "217273",   # ← 이걸로 보내면 0건 응답
}
```

**2026-06-03 실측 결과**: 위 코드는 HTTP 200 + `[]` (0건) 응답함. 데이터가 없는 게 아니라 **파라미터명이 틀려서 서버가 빈 결과를 반환**하는 것.

**올바른 파라미터명 (소문자)**:

```python
# ✅ 실측 동작 확인
params = {
    "REQ_DAT_FR": "20260601",
    "REQ_DAT_TO": "20260630",
    "cst_cod": "217273",        # ← 소문자
}
# HTTP 200 + 2,592건 응답 (6/1~6/30 한 달치, P217273 유원피에스)
```

## 검증 재현 절차

같은 세션에서 4가지 호출을 비교하면 결정적:

| # | URL params | 응답 |
|---|------------|------|
| 1 | `REQ_DAT_FR/TO=20260601~20260630&cst_cod=217273` | **2,592건** ✅ |
| 2 | `REQ_DAT_FR/TO=20260601~20260630&STC_CST_COD=217273&REQ_CST_COD=217273` | `[]` (0건) ❌ |
| 3 | `REQ_DAT_FR/TO=20260601~20260630&REQ_CST_COD=217273` (대문자 하나만) | `[]` (0건) ❌ |
| 4 | `REQ_DAT_FR/TO=20260601~20260630` (고객사 파라미터 생략) | 0건 (전체 DB 기준, 거래처 무관) ❌ |

→ 고객사 필터는 **반드시 `cst_cod` (소문자)**. 대문자 변형은 모두 0건 응답.

## 왜 이런 함정이 생겼나

스킬 본문의 "기존 데이터 조회" 섹션이 PBM110MW 신규등록 페이로드(스킬 본문 line 120~ 부근, 소문자 `cst_cod` 아님 `CST_COD` 등 다른 컨벤션)와 `fn_search` HTML 폼 필드명(HTML은 `id="STC_CST_COD"` 형태)을 **혼동**해서 표기된 것으로 추정됨. 하지만 WPPS의 `/ps/PBM110MW.do` GET 엔드포인트는 **Tomcat 컨트롤러 시그니처에서 소문자 파라미터**를 받음.

## 영향 범위

- **PBM110MW 조회 전용** (read-only). 저장 시 신규 페이로드는 `CST_COD` 필드가 아니라 `ARV_CST_COD`(도착지 고객사) 등 다른 필드를 사용하므로 영향 없음.
- PBM140MW (출하통보)는 별도 API라 본 함정 무관. `kpp-verify-search.py` 등 PBM140MW 스크립트는 `dlv_dat`/`arv_dat` 등 소문자 컨벤션 일관 사용.

## 조치

- 위임용 지시서(`/opt/hermes/plans/kpp-2026-06-unload-check.md`)에는 이미 `cst_cod` (소문자)로 정정 반영 완료.
- SKILL.md 본문 패치는 후속 작업에서 진행 예정 (검토 모드라 직접 도구 접근 불가, 일반 세션에서 패치 권장).

## 교훈

KPP 스킬의 "기존 데이터 조회" 예제를 그대로 복사해서 쓰면 **무조건 0건** 응답함 → 사용자가 "데이터가 없다"고 오해할 위험. **반드시 `cst_cod` (소문자) 사용**을 명심.
