## 핵심 원칙

### Chesterton's Fence
> 왜 있는지 모르는 코드는 삭제하기 전에 반드시 이유를 파악하라

- 레거시 코드 삭제 전, 왜 그렇게 작성되었는지 먼저 이해
- 의도를 모르면 삭제하지 말고 주석으로 질문 남기기
- 삭제 후 예상치 못한 부작용 발생 시 즉시 복구

### Rule of 500
> 함수/컴포넌트는 500줄 이하로 유지

- 500줄 넘으면 분리 검토
- production-plan.tsx (1400줄+) → 컴포넌트 분리 필요
- 각 함수는 하나의 책임만

### 단순화 체크리스트
1. 동일한 동작을 더 적은 코드로 할 수 있는가?
2. 중복 로직이 있는가?
3. 불필요한 추상화가 있는가?
4. 이름이 의도를 명확히 표현하는가?
5. 복잡도를 줄이면서 동작이 보존되는가?
