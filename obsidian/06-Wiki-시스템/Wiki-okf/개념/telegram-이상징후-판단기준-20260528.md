# Telegram 크론잡 이상징후 판단 기준

> **목적:** Telegram으로 전송되는 30분 간격 감시 보고서에서 "이상징후"로 표시되는 항목 중
> 실제 심각한 문제와 정상 동작을 구분하기 위한 기준 문서.
> 
> **적용 대상:** `ki-ai-trader 실시간 감시 (장중 30분)` 크론잡 (job_id: 180225f12900)

## 1️⃣ 정상 동작이나 로그상 "에러"로 보이는 것

### HTTP 429 (Too Many Requests)

| 항목 | 내용 |
|:----|:-----|
| **크론잡 판단** | ⚠️ 에러 (일시적) |
| **실제 심각도** | ✅ **정상 — rate limit 도달, 자동 복구됨** |
| **발생 빈도** | 하루 1~2회 (오전/오후 각 1회) |
| **지속 시간** | 수 초 이내 (일시적) |
| **영향** | 해당 API 호출만 실패, 다음 루프에서 재시도 |

**판단 근거:**
- Kiwoom API는 초당/분당 호출 제한이 있음 (명시적 문서화되지 않음)
- 시세 조회(`get_stock_price`)와 ATR 데이터 요청이 5초 간격으로 동시에 발생할 때 rate limit 도달
- HTTPS 429는 HTTP 표준 응답 — 서버가 "잠깐만 쉬어라"고 알리는 것
- 재시도 없이 자동 복구되며 트레이딩 로직에 누적 영향 없음
- **지속적으로 증가(3회/일→10회/일)하지 않으면 무시 가능**

**False Positive 조건:**
```
if 429_occurrences_per_day <= 3 → ✅ 정상 (일시적 과부하)
if 429_occurrences_per_day > 10  → ⚠️ 모니터링 필요 (속도 제어 검토)
```

### 체결 타임아웃 (매수/매도 후 체결 미확인)

| 항목 | 내용 |
|:----|:-----|
| **크론잡 판단** | 🔴 에러 (4/4건) |
| **실제 심각도** | ✅ **정상 — 실제 체결은 완료됨** |
| **발생 빈도** | 신규 주문 시 항상 발생 |
| **영향** | 없음 (잔고 정상 반영) |

**판단 근거:**
- Kiwoom API에 주문 접수(`return_code=0`) → 실제 체결은 즉시 이루어짐
- 시스템이 체결 확인을 위해 30초/60초 후 폴링하지만, 체결 시점이 폴링 직후라서 놓침
- 이후 포트폴리오 업데이트 시 잔고는 정상 반영됨
- `로그: "매도 체결 미확인"` 이후 `로그: "보유 종목 없음: 084990"` → 실제로는 매도 완료됨
- **트레이딩 의사결정에 영향 0%**

**False Positive 조건:**
```
if 체결_실패_후_잔고_정상_반영됨 → ✅ 정상 (폴링 시차 문제)
if 체결_실패_후_잔고_미반영_+_재시도_초과 → 🔴 실제 장애
```

### LLM 호출 실패 ('NoneType' object has no attribute 'strip')

| 항목 | 내용 |
|:----|:-----|
| **크론잡 판단** | 🔴 에러 (208회) |
| **실제 심각도** | ⚠️ **fallback 정상 동작 중** |
| **영향** | 시그널 품질 일부 저하 가능 |

**판단 근거:**
- LLM(OpenRouter/AI 공급자) 응답 파싱 과정에서 None 반환
- `_call_llm()` 메서드에서 response가 예상된 형식이 아닐 때 발생
- **Multi-Agent consensus는 자체 분석(quant/sentiment/macro/memory 4개 Agent)으로 정상 생성**
- `[INFO] Analysis posted: quant_agent...` 로그는 정상 출력 중
- `[INFO] [종목명] Multi-Agent consensus: BUY/HOLD/SELL` — fallback 결정 로직 정상
- **에러 횟수(208회) ≠ 영향도 — 단순 LLM evaluate 실패일 뿐**

**False Positive 조건:**
```
if fallback_consensus_정상_출력 → ✅ 경고만 (fallback 유지)
if fallback_consensus_실패 + 시그널_미생성 → 🔴 실제 장애
```

### 주문가능금액 부족

| 항목 | 내용 |
|:----|:-----|
| **크론잡 경고** | 현금 315,066원 중 94,519원만 사용 가능 |
| **실제 심각도** | ⚠️ **구조적 문제 — 설계 한계** |

**판단 근거:**
- `MAX_POSITION_PCT = 0.50` (자본금의 50% = 매수한도)
- ATR 손절가를 고려한 40% 추가 버퍼
- 실현: 315,066 × 0.50 = 157,533원 → 버퍼 적용 후 ≈ 94,519원
- 고가주(삼성전자 80k+, SK하이닉스 200k+)는 1주도 매수 불가
- **기능적 버그가 아니라 자본금 규모 대비 설계 한계**

**False Positive 조건:**
```
if 현금_증가_추세 + 고가주_미매수 → ⚠️ 구조적 한계 (버그 아님)
if 현금_충분 + 매수_안_됨 → 🔴 실제 버그
```

## 3️⃣ 크론잡 판단 개선 요약

| 이상징후 | 크론잡 표시 | 실제 판단 | 조치 필요? |
|:---------|:----------:|:---------:|:----------:|
| HTTP 429 (1~2회/일) | ⚠️ 에러 | ✅ 정상 | ❌ |
| 체결 타임아웃 | 🔴 에러 | ✅ 정상 | ❌ |
| LLM NoneType.strip() (208회) | 🔴 에러 | ⚠️ 경고 (fallback 유지) | 추후 |
| 헬릭스미스 손절 | ℹ️ 정보 | ✅ 정상 (수정 완료) | ❌ |
| 매수 연속 실패 (8회) | ⚠️ 주의 | ⚠️ 구조적 한계 | 버퍼 조정 검토 |
| 주문가능금액 부족 | ⚠️ 주의 | ⚠️ 구조적 한계 | 자본금 증액/버퍼 조정 |

**결론:** 현재 Telegram 보고서의 "에러" 표시는 대부분 **정상 동작의 로그 노이즈**입니다.
실제 트레이딩에 영향을 주는 문제는 **자본금 부족(매수 불가)** 하나뿐입니다.
