## 1. 생산계획 테이블 컬럼 (13열)

### 헤더 순서 (엑셀 일치)

| # | 헤더명 | DB 필드명 | 데이터 타입 | 설명 | 표시 예시 |
|:-:|:-------|:----------|:-----------|:-----|:---------|
| 1 | 상태 | `status` | string | pending/started/ended/stopped | 대기 |
| 2 | 일자 | `date` | date | YYYY-MM-DD | 2026-06-23 |
| 3 | 기계 | `machine_number` | string | 기계 번호 | 3 |
| 4 | 금형 | `mold_number` | string | 금형 번호 | 122 |
| 5 | 제품명 | `product_name` | string | 한글 제품명 | 탑백 52L |
| 6 | 영문명 | `product_name_eng` | string | 영문 제품명 | Top Bag 52L |
| 7 | 색상 | `color1` | string | 색상 코드 (영문) | WHITE1 |
| 8 | 롯트번호 | `color2` | string | 색상 코드2 | WHITE 180 |
| **9** | **단위** | **`unit_quantity`** | int | **팔레트/BOX당 낱개 수** | **125개** |
| **10** | **생산수량** | **`quantity`** | int | **생산할 단위(P/BOX) 개수** | **1** |
| **11** | **생산단위** | **`unit`** | string | **생산 단위 종류** | **P** |
| 12 | 작업예정 수량(낱개) | `total` | int | = 단위 × 생산수량 | 125 |
| 13 | 작업 | — | — | 버튼 영역 | — |

### 핵심 (9~11열)

```
9열(단위) = unit_quantity     → "125개"  (팔레트당 125개)
10열(생산수량) = quantity     → "1"     (1팔레트 생산)
11열(생산단위) = unit         → "P"     (팔레트 단위)
12열(작업예정) = total        → "125"   (= 1 × 125)
```

### 헷갈리지 말 것

| 잘못된 생각 | 올바른 생각 |
|:-----------|:-----------|
| 단위(9열)는 "P" 또는 "BOX"이다 | ❌ **단위(9열)는 "125개"처럼 낱개 수를 표시** |
| 생산단위(11열)는 숫자(낱개 수)이다 | ❌ **생산단위(11열)는 "P" 또는 "BOX" 같은 단위 종류** |
| 생산수량(10열)이 단위(9열)보다 클 수 있다 | ❌ **단위 ≥ 생산수량이 정상. 반대면 SWAP 의심** |

## 2. DB unique 제약 조건

```sql
-- 2026-06-23 적용
uniq_productionlog_full_key:
  (date, machine_number, mold_number, product_name, color1, color2,
   unit, quantity, unit_quantity)
```

- 동일한 (날짜+기계+금형+제품+색상+단위+수량) 조합은 중복 불가
- 단위(unit_quantity)와 생산수량(quantity)이 다르면 별도 행 허용

## 3. 날짜 이월

- `POST /api/production-log/carry-forward?from_date=어제&to_date=오늘`
- 전일 `status != 'ended'` 항목 → 금일로 date 변경
- 중복 방지: to_date에 동일 키 조합이 존재하면 SKIP
- ended(완료) 작업은 원래 날짜 유지
