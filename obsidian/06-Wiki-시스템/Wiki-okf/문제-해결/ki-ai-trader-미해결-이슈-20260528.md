## 🔴 심각 — 즉시 조치 권장

### Issue 1: LLM 호출 실패 (NoneType.strip())

| 항목 | 값 |
|:----|:---|
| **발생 빈도** | 80~208회/일 (지속 증가 중) |
| **에러 메시지** | `'NoneType' object has no attribute 'strip'` |
| **발생 위치** | LLM 응답 파싱 과정 |
| **최초 발견** | 2026-05-28 08:21 이후 지속 |
| **어제 (5/27) 대비** | 4,240회 (HTTP 401 Unauthorized) → 오늘은 유형 변경됨 |

**증상:**
- 거의 모든 AI 평가 사이클에서 LLM 호출 실패
- Agent 분석(quant/sentiment/macro/memory)은 정상 생성
- Multi-Agent consensus는 fallback 로직으로 정상 출력 중
- 그러나 LLM 기반 트레이딩 의사결정 품질이 저하되었을 가능성 높음

**영향:**
- ⚠️ 시그널 퀄리티 저하
- ⚠️ 시스템이 fallback에 의존하게 되어 시장 변화 대응력 약화

**분석:**
- 어제는 HTTP 401 (인증 실패) → 오늘은 NoneType.strip() (응답 파싱 실패)
- 유형이 변경된 것으로 보아 LLM 공급자(Ollama/Z-AI) 측 문제 가능성
- `/api/dashboard/ai-status` 엔드포인트 자체도 `'settings' is not defined` 오류 반환 중

### Issue 2: 체결 타임아웃 전수 발생

| 항목 | 값 |
|:----|:---|
| **발생 건수** | 오늘 4/4건 (매수 2건, 매도 2건) |
| **타임아웃** | 30초 → 60초 후 체결 미확인 |
| **발생 시간** | 09:04, 09:08, 09:20, 09:27 |

**증상:**
- 모든 주문이 `return_code=0` (정상 접수)이나 30초·60초 내 체결 확인 실패
- `매도 체결 미확인` 로그에도 불구하고 잔고는 정상 반영됨 (실제 체결은 됨)
- API 폴링이 체결 시점보다 늦은 것으로 추정

**영향:**
- ⚠️ 시스템이 "체결 미확인" 상태로 로직 진행 → 향후 정합성 문제 유발 가능
- ⚠️ 매도 후 재매수 타이밍 지연 가능성

**분석:**
- Kiwoom API 체결 확인 폴링 간격 또는 타임아웃 임계값 문제
- 실제 체결은 정상이므로 API 조회 주기 또는 재시도 로직 개선 필요

## 🟡 중간 — 모니터링 필요

### Issue 3: DB accounts 테이블 누락

| 항목 | 값 |
|:----|:---|
| **오류** | `sqlite3.OperationalError: no such table: accounts` |
| **발생** | 2026-05-28 00:30 (1회), 이후 재발 없음 |
| **영향** | dashboard/summary의 account 객체가 `{}` (빈 객체)로 리턴 |

**증상:**
- SQLite DB에 `accounts` 테이블이 존재하지 않음
- DB sync 실패로 계좌 정보(현금 잔고, 평가금, 주문가능금액)가 API에서 제공되지 않음
- 로그에서 `포트폴리오 업데이트: 현금 236,899원`은 정상 출력 → 거래 자체에는 영향 없음

**분석:**
- 마이그레이션 누락 가능성 (초기 DB 스키마 생성 시 accounts 테이블이 포함되지 않음)
- `src/database/` 디렉토리 마이그레이션 파일 확인 필요

### Issue 4: 자본금 부족으로 인한 고가주 매수 실패 패턴

| 항목 | 값 |
|:----|:---|
| **주문가능금액** | 약 64,647원 → 약 118,449원 (변동) |
| **매수 시도 실패** | 삼성전자(303~307k), 한화오션(134.9k), SK이노베이션(115.4k), 셀트리온(194.5k), 삼성에스디에스(247.5k) |
| **발생 횟수** | 8회 이상 (09:04~09:27 사이 집중) |

**증상:**
- 매수 후보 선정 시 현금 보유량을 고려하지 않고 고가 종목을 추천
- 1주 가격이 주문가능금액을 초과하는데도 반복적으로 매수 시도
- 매수 실패 자체는 시스템에 무해하나, 트레이딩 사이클 낭비

**분석:**
- 조건8 (`price > 100000 → continue`)이 이미 10만원 이상 종목을 제외
- 그러나 10만원 미만 필터링을 통과하더라도 현금 64k~118k로 살 수 있는 종목이 제한됨
- **매수 후보 선정 로직에 현금 대비 적정 가격대 필터링 필요**

### Issue 5: 토큰 리프레시 문제

| 항목 | 값 |
|:----|:---|
| **오류** | `⚠️ 리프레시 토큰 없음 - 새로 인증` |
| **발생 빈도** | 하루 3~4회 (08:00, 08:30, 08:53 등) |
| **영향** | 매번 client credentials 재인증 (Bearer token, 만료 1시간) |

**증상:**
- refresh token이 저장되지 않아 매번 OAuth client credentials flow로 재인증
- 재인증 자체는 정상 동작하지만 불필요한 인증 오버헤드 발생

**분석:**
- 토큰 캐시 파일 (`.token_cache`)은 존재하나 refresh token이 저장되지 않음
- token refresh 로직에서 refresh token을 캐시에 쓰는 코드 누락 가능성

### Issue 6: 종목코드 오류 (없는 종목 조회)

| 항목 | 값 |
|:----|:---|
| **오류** | `[ERROR] ❌ 보유 종목 없음: 084990` |
| **발생** | 로그에서 1회 확인 |

**증상:**
- 존재하지 않거나 이미 매도된 종목코드(084990)를 보유 종목으로 간주하고 조회 시도
- 헬릭스미스(084990) 매도 후 잔여 참조 문제로 추정

**분석:**
- 매도 완료 후 포트폴리오 업데이트 시 종목 리스트에서 완전히 제거되지 않음
- 포트폴리오 정리(cleanup) 로직 점검 필요

### Issue 7: 계좌 API account 필드 공백

| 항목 | 값 |
|:----|:---|
| **엔드포인트** | `/api/dashboard/summary` |
| **증상** | `account` 필드가 `{}` (빈 객체) |
| **영향** | Cron 보고서에서 계좌 잔고 정보 부재 |

**증상:**
- 대시보드 API의 `account` 객체가 항상 빈 값으로 반환
- 현금 잔고, 평가금, 주문가능금액이 API 레벨에서 제공되지 않음
- 다만 로그 레벨(`포트폴리오 업데이트`)에서는 정상 출력

**분석:**
- Issue 3 (accounts 테이블 누락)과 연관 가능성 높음
- account 정보를 DB에서 읽어오는 로직이 실패하면서 빈 객체 반환
