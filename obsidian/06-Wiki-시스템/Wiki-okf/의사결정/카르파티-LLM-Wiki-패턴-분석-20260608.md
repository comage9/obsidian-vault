## 1. 카르파티 LLM Wiki의 핵심 아이디어 (요약)

### 문제 정의: RAG의 한계

> "Most people's experience with LLMs and documents looks like RAG: you upload a collection of files, the LLM retrieves relevant chunks at query time, and generates an answer. This works, but the LLM is rediscovering knowledge from scratch on every question. There's no accumulation."

→ **RAG = 매번 처음부터 재발견, 누적 없음.** 이게 곰너이 님이 짚으신 "매번 처음 대하듯 행동하는 문제"의 산업 표준 명명입니다.

### 해결: Persistent Wiki

> "Instead of just retrieving from raw documents at query time, the LLM **incrementally builds and maintains a persistent wiki** — a structured, interlinked collection of markdown files that sits between you and the raw sources."

**핵심 차이:**
- ❌ RAG: 매 query마다 raw document에서 chunk retrieval
- ✅ LLM Wiki: **한 번 통합/요약/링크**된 persistent markdown 컬렉션이 raw source와 user 사이에서 작동

> "The wiki is a persistent, compounding artifact. The cross-references are already there. The contradictions have already been flagged. The synthesis already reflects everything you've read."

## 2. 카르파티의 3-Layer 아키텍처

```
┌─────────────────────────────────────────────┐
│  Layer 1: Raw Sources (불변)                 │
│  - Articles, papers, images, data files      │
│  - LLM이 읽기만 하고 절대 수정 안 함         │
│  - Source of truth                           │
├─────────────────────────────────────────────┤
│  Layer 2: The Wiki (LLM이 모두 소유/관리)    │
│  - LLM-generated markdown 파일들             │
│  - Summaries, entity pages, concept pages    │
│  - Cross-references, comparisons, synthesis  │
│  - 사람은 읽기만, LLM은 쓰기/유지보수 전부   │
├─────────────────────────────────────────────┤
│  Layer 3: The Schema (설정 파일)             │
│  - CLAUDE.md (Claude Code) / AGENTS.md       │
│  - 위키 구조, 컨벤션, workflow 규칙          │
│  - LLM을 "disciplined wiki maintainer"로 만듦│
│  - 사람과 LLM이 공동 진화시킴                │
└─────────────────────────────────────────────┘
```

**"Obsidian은 IDE, LLM은 프로그래머, 위키는 코드베이스"** — 카르파티의 비유.

## 3. 핵심 Operations 3가지

### A. Ingest (수집)
- 새 source를 raw 폴더에 넣고 LLM에게 처리하라고 지시
- LLM은: 읽기 → 핵심 정리 요약 → entity/concept 페이지 업데이트 → cross-reference 갱신 → log 기록
- **한 source가 10~15개 wiki 페이지를 건드릴 수 있음**

### B. Query (질의)
- 사용자가 질문 → LLM이 관련 페이지 검색 → 합성 + 인용
- 답변 형태: markdown 페이지, 비교표, 슬라이드(Marp), 차트, 캔버스
- **핵심 통찰:** 좋은 답변을 다시 wiki에 새 페이지로 filed back → 대화 내용도 compound됨

### C. Lint (정기 점검)
- 모순 페이지, 오래된 claim, orphan 페이지 (inbound link 없음), 중요한 concept인데 페이지 없음, cross-reference 누락, web search로 채울 수 있는 data gap
- LLM이 새 질문/탐구 제안

## 4. Indexing & Logging (필수 두 파일)

### `index.md` (내용 지향)
- 위키 전체 카탈로그 — 각 페이지 + 1줄 요약 + 메타데이터(날짜, source 수)
- 카테고리별 조직 (entities, concepts, sources 등)
- **매 ingest 시 LLM이 업데이트**
- Query 시: index 먼저 읽고 → relevant page drill-in
- **중간 규모(~100 sources, 수백 페이지)에서 RAG embedding 없이 놀라울 정도로 잘 작동**
- **"~100 sources를 넘어가면 임베딩 기반 RAG 인프라 필요 없어도 됨"**

### `log.md` (시간순)
- Append-only 기록: ingest, query, lint pass
- `## [2026-04-02] ingest | Article Title` 같은 일관된 prefix
- Unix 도구로 parseable: `grep "^## \[" log.md | tail -5` → 최근 5개 entry
- 위키 진화 timeline + 최근 작업 컨텍스트 제공

## 7. 곰너이 님 시스템이 이미 갖고 있는 카르파티 패턴 이상의 것

1. **크로스 디바이스 (Syncthing)** — 카르파티는 single-host, 곰너이 님은 Windows + Linux 자동 동기화
2. **3중 검증 의무 (Wiki + README + Git push)** — 카르파티는 wiki만, 곰너이 님은 3-tier
3. **핸드오프 노트** — 컨텍스트 컴팩션 시 핵심 보존, 카르파티는 언급 없음
4. **자가학습 cron 5개** — Lint가 자동화됨 (매일 밤)
5. **Telegram/Discord 메시징 통합** — 카르파티는 로컬 LLM만, 곰너이 님은 멀티 채널
6. **절대 금지/각서 박제** — 실패가 절대 금지 #N으로 영구 보존

## 8. 카르파티 패턴 대비 곰너이 님 시스템의 격차 (해결 필요)

### 1순위: **자동 recall 부재** (곰너이 님이 직접 짚으신 핵심)
- 카르파티: query 시 LLM이 index.md 자동으로 읽고 relevant page drill-in
- 곰너이 님: **에이전트가 자발적으로 search_files 호출해야 함**
- → 매번 처음 대하듯 행동하는 문제의 직접 원인

### 2순위: **index.md 자동 갱신 부재**
- 카르파티: 매 ingest 시 index.md 자동 업데이트
- 곰너이 님: index 파일 자체가 없거나 수동 갱신
- → 검색 품질/속도 저하

### 3순위: **log.md (append-only timeline) 없음**
- 카르파티: `## [date] action | title` 형식 markdown
- 곰너이 님: session_search DB (binary, markdown 아님)
- → Git으로 grep/diff 가능, 사람이 읽기 좋음

### 4순위: **qmd 같은 hybrid search 도구 없음**
- 카르파티: BM25 + vector + LLM re-ranking on-device
- 곰너이 님: 단순 ripgrep
- → 위키가 수백 페이지를 넘어가면 검색 품질 저하

### 5순위: **Ingest 시 10-15 페이지 동시 갱신 패턴 미확립**
- 카르파티: 1 source = 10-15 page 갱신
- 곰너이 님: 보통 1-2개 문서 갱신
- → 위키 풍부화 속도 차이
