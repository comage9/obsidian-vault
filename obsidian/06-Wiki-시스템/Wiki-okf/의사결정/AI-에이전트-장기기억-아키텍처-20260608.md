## 1. 장기기억의 세 가지 표준 분류 (Redis 블로그, 2026-04)

Redis의 "Long-Term Memory Architectures for AI Agents" 블로그에서 정리한 인지과학 기반 분류 — **대부분의 production 시스템이 이 세 가지를 결합**:

| 유형 | 정의 | 예시 |
|------|------|------|
| **Semantic memory** (의미 기억) | 시간/맥락과 무관한 사실, 개념, 요약 | 사용자 선호, 도메인 규칙, distilled summaries |
| **Episodic memory** (에피소드 기억) | 시간 인덱싱된 경험, 사건 | 특정 대화, tool call, timestamp가 있는 로그 |
| **Procedural memory** (절차 기억) | 스킬, 루틴 | 프롬프트, 정책, agent 코드 |

**핵심 패턴:** Episodic memory는 시간이 지나면서 Semantic memory로 consolidation(통합/요약)됨.

출처: https://redis.io/blog/long-term-memory-architectures-ai-agents/

## 2. 표준 아키텍처: "Read-Before-Reasoning, Write-After-Acting" 루프

Redis 블로그가 정리한 production agent의 표준 루프:

```
1. Recall: 현재 working memory로 relevant memory pull
2. Reason: LLM이 작업 수행
3. Act: tool 호출, 응답 생성
4. Update: 새 사실을 long-term store에 추출, old context 요약
5. Loop: 다음 입력으로 반복
```

**가장 어려운 부분 = Context Assembly**: "컨텍스트 윈도우에 들어갈 수 있는 모든 것 중에서 실제로 무엇을 넣을지" 결정. → 이것이 곰너이 님 시스템에서 mandatory-verification Step 1-2 (Wiki 검색 + 스킬 로드)의 정체성.

## 6. 곰너이 님 시스템의 강점 (표준 대비)

1. **Git 기반 영속성** — 표준 시스템은 vector DB/SaaS 의존, 곰너이 님은 obsidian + git으로 **완전 자체 호스팅 + 버전 관리**
2. **크로스 디바이스 동기화** — Syncthing으로 Windows ↔ Linux 자동 동기화. 표준 시스템은 single-host.
3. **3중 검증 의무** — Wiki + README + Git push. 표준은 보통 1-tier (DB + retrieval).
4. **절대 금지/각서 박제** — 실패를 "절대 금지 #N"으로 영구 보존. 표준은 보통 soft warning.
5. **핸드오프 노트** — 컨텍스트 컴팩션 시 핵심 정보 손실 방지. 표준 시스템은 보통 컨텍스트 손실.

## 7. 곰너이 님 시스템의 약점 (표준 대비, 곰너이 님이 정확히 짚으신 부분)

1. **검색이 자동화되지 않음** — 표준 mem0/Letta는 매 task 시작 시 자동으로 relevant memory recall. 곰너이 님 시스템은 **에이전트가 자발적으로 `search_files` 호출해야 함**. → 매번 처음 대하듯 행동하는 문제의 직접 원인.
2. **Importance scoring 없음** — MEMORY.md 99% 포화 시 모든 항목이 동등한 우선순위. 표준은 recency/importance 점수 기반 자동 정리.
3. **Retrieval 품질** — `search_files`는 단순 grep 기반. 표준은 vector + full-text + temporal 융합 (RRF 등).
4. **Consolidation 자동화** — episodic → semantic 변환이 수동. 표준은 자동 요약/추출.
5. **Conflict resolution** — 두 MEMORY 항목이 충돌할 때 자동 감지 없음.
6. **Time-decay** — 오래된 항목 자동 만료 없음. 잘못된 정보가 영구 보존될 수 있음.

## 9. 곰너이 님의 의도 추측 (구체적 제안)

곰너이 님이 "확인이 안 된다"고 지적하신 문제 → 표준 시스템의 "Read-Before-Reasoning" 루프가 곰너이 님 시스템에 **자동화되어 있지 않기 때문**으로 추정됨.

**구체적 해결 방향 (제안)**:

### A. 검색 자동화 (가장 시급)
- 매 사용자 지시 수신 시 **자동으로 1차 검색** 트리거
- 키워드 추출 → `search_files`로 Wiki 검색 → 결과가 있으면 절차 따르기
- 결과가 없을 때만 처음 대하듯 진행

### B. Importance/Recency 점수 도입
- MEMORY 항목에 "마지막 사용일" 또는 "중요도" 태그
- 90% 포화 시 가장 오래되고 덜 사용된 항목부터 자동 후보

### C. Vector Search 추가
- obsidian vault에 대한 vector embedding 인덱스
- 단순 grep → 의미 기반 검색으로 upgrade

### D. Consolidation 자동화
- session_search 결과를 주기적으로 Wiki로 요약/이동
- 핸드오프 노트의 핵심 결정사항을 자동으로 MEMORY에 추출

## 10. 결론 (한 줄)

**곰너이 님이 구축한 시스템은 표준 long-term memory 아키텍처의 핵심(3-tier 분류, write-after-acting, 3중 검증)을 대부분 갖추고 있으나, "자동 recall"이 빠져 있어서 매번 수동 검색 의존도가 높음.** 이것이 "매번 처음 대하듯 행동하는 문제"의 직접 원인일 가능성이 큼. 표준 시스템(mem0, Letta) 참고 시 **자동 recall + importance scoring** 두 가지를 1순위로 도입 검토 권장.
