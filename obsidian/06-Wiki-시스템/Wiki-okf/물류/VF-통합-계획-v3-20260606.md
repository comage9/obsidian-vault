# VF 바코드 시스템 ↔ VF 프로젝트 통합 계획 v3

> **기반:** v2 계획 + Harness 컨설팅 (M3 MiniMax + DeepSeek-V4-flash 교차 분석)
> **핵심 변경:** 단일 테이블 + source 컬럼, 멱등성 키 도입, Google Sheets 단방향 동기화

---

## 1. 핵심 목적 (변경 없음)

**VF 바코드 = 입력 데이터를 바코드로 생성하는 메인 인터페이스**

- 생성 바코드 2종: (1) 입력 데이터 기반 바코드 (2) 송장 번호 바코드
- VF 바코드 입력 데이터 = 그날의 최종 출고 수량으로 VF 프로젝트에 반영
- 기존 VF 프로젝트 품목 데이터는 그대로 활용 (공유 DB)

---

## 2. 데이터 모델 (컨설팅 반영)

### 변경 사항 요약

| 항목 | v2 (기존) | v3 (컨설팅 반영) | 출처 |
|:-----|:---------|:----------------|:-----|
| 테이블 구조 | 2개 분리 (vf_delivery_items + barcode_input_items) | **단일 테이블 + source 컬럼** (우선) 또는 2개 테이블 유지+멱등성 키 | DS 권장 |
| 멱등성 | 미정의 | **idempotency_key UNIQUE** 필수 | M3+DS 일치 |
| 제품 정보 | 각 테이블에 중복 저장 | **별도 products 테이블**로 정규화 | M3+DS 일치 |
| 취소/정정 | 미정의 | **is_cancelled + 취소 이벤트** 설계 | M3 제안 |
| Google Sheets | 양방향 동기화 | **단방향(DB → Sheets)** 백업 전용 | DS 권장 |

### 제안 1: 단일 테이블 + source 컬럼 (우선 권장)

```sql
-- 제품 마스터 테이블 (신규)
CREATE TABLE products (
    product_code TEXT PRIMARY KEY,
    product_name TEXT NOT NULL,
    category TEXT,
    is_active BOOLEAN DEFAULT TRUE,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 통합 출고 데이터 테이블
CREATE TABLE delivery_items (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    
    -- 멱등성 키 (중복 입력 방지)
    idempotency_key TEXT UNIQUE NOT NULL,  -- 송장번호 + date 조합 해시
    
    -- 비즈니스 데이터
    product_code TEXT NOT NULL REFERENCES products(product_code),
    quantity INTEGER NOT NULL CHECK (quantity > 0),
    delivery_date DATE NOT NULL,
    time_slot TEXT,
    barcode_data TEXT,
    tracking_no TEXT,
    
    -- 출처 (데이터 구분)
    source TEXT NOT NULL DEFAULT 'barcode' 
        CHECK (source IN ('vf_project', 'barcode', 'correction')),
    
    -- 메타데이터
    input_user_id TEXT,
    is_cancelled BOOLEAN DEFAULT FALSE,
    cancelled_at DATETIME,
    cancel_reason TEXT,
    
    -- 타임스탬프
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    
    -- 인덱스
    UNIQUE(delivery_date, product_code, time_slot, source),
    INDEX idx_delivery_date (delivery_date),
    INDEX idx_product_code (product_code),
    INDEX idx_tracking_no (tracking_no)
);

-- 통합 조회 뷰 (취소 제외한 유효 데이터만)
CREATE VIEW vw_daily_delivery_total AS
SELECT 
    delivery_date,
    product_code,
    p.product_name,
    p.category,
    SUM(quantity) as total_quantity,
    COUNT(*) as record_count
FROM delivery_items d
JOIN products p ON d.product_code = p.product_code
WHERE d.is_cancelled = FALSE
GROUP BY delivery_date, product_code;
```

### 제안 2: 2개 테이블 유지 (최소 변경, v2 유지 시)

v2案 유지 시 아래 필수 추가:
- 각 테이블에 `idempotency_key TEXT UNIQUE` 추가
- `source TEXT DEFAULT 'project'/'barcode'` 추가
- FK 제약조건으로 제품 코드 유효성 검증
- 뷰에 `is_cancelled` 조건 추가

## 3. 통합 3단계 (Phase 조정)

### Phase A: React 컴포넌트화 ⏱ 2-3시간 (변경 없음)

| 작업 | 내용 |
|:-----|:------|
| A-1 | `pages/barcode-generation.tsx` 생성 |
| A-2 | 상태 관리 통합 (`useDeliveryData` 훅) |
| A-3 | 사이드바 교체 (`externalUrl` → React Router) |
| A-4 | 제품 DB 통합 (바코드 페이지 → products 테이블 조회) |

### Phase B: 데이터 통합 ⏱ 3-4시간 (컨설팅 반영으로 확장)

| 작업 | 내용 | 상세 |
|:-----|:------|:-----|
| B-1 | **DB 스키마 마이그레이션** | products 테이블 생성, delivery_items 통합 또는 기존 테이블에 컬럼 추가, idempotency_key 도입 |
| B-2 | **데이터 이관** | 기존 vf_delivery_items 데이터를 products + delivery_items로 이관 (source='vf_project') |
| B-3 | **멱등성 처리 API** | `POST /api/delivery/hourly` — idempotency_key로 중복 체크 후 INSERT or SKIP |
| B-4 | **통합 조회 API** | `GET /api/delivery/total?date=` — 취소 제외 유효 데이터 합산 반환 |
| B-5 | **취소/정정 API** | `POST /api/delivery/cancel` — is_cancelled 플래그 설정 |
| B-6 | **Google Sheets 단방향 동기화** | DB→Sheets 백업 (5분 배치, 실패 시 재시도) |

### Phase C: 고도화 ⏱ 선택

| 작업 | 내용 |
|:-----|:------|
| C-1 | 실시간 바코드 스캔 연동 (웹캠/모바일) |
| C-2 | 인쇄 템플릿 커스터마이징 |
| C-3 | 출고 데이터 → 생산 계획 자동 반영 |

## 4. 데이터 흐름 (최종)

```
[Google Sheets]
    ↕ (단방향 백업, 5분 배치)
[공유 DB] ─── products (제품 마스터)
              └── delivery_items (통합 출고 데이터)
                    ├── source='vf_project' (기존 데이터)
                    ├── source='barcode' (신규 바코드 입력)
                    └── source='correction' (정정 데이터)
    ↕
[VF 프로젝트] ←── vw_daily_delivery_total (통합 조회)
    ↕
[VF 바코드 페이지] ←── 입력 → POST /api/delivery/hourly (멱등성 보장)
```

## 5. 리스크 (컨설팅 반영)

| 리스크 | 대응 | 컨설팅 출처 |
|:-------|:-----|:-----------|
| 동일 idempotency_key 중복 요청 | UNIQUE 제약 + HTTP 409 Conflict 반환 | M3+DS |
| Google Sheets 동기화 실패 | 재시도 로직 + 실패 로그 알림 | DS |
| 제품 코드 변경 시 과거 데이터 정합성 | products 테이블 정규화 + 코드 변경 시 이력 관리 | M3+DS |
| 바코드 중복 스캔 | idempotency_key로 1차 방어 + 프론트엔드 중복 체크 | M3 |
| 취소 이후 재입력 | is_cancelled 해제 또는 새 idempotency_key 발급 | M3 |
| DB 마이그레이션 중 데이터 소실 | 데이터 이관 전 백업 + dry-run 테스트 | DS |
