# KPP 조회 재시도 패턴 (2026-06-04 발견)

## 문제
Edge 브라우저 첫 신규 탭에서 PBM140MW 조회 시:
- 1차 search 클릭 + 8초 대기 → **행수 0** (잘못된 결과)
- 재시도(같은 search 버튼 2회 클릭) + 8초 대기 → **정상 표시**

Chrome에서는 1회 조회에 바로 표시되던 것과 차이점.

## 원인 추정
- SpreadJS가 `sr_dlv_dat_f/t` input change 이벤트로 검색 조건 변경 → DB 재조회 → 그리드 렌더링
- 첫 클릭은 서버 호출만 보내고, 클라이언트 캐시 미스 + SpreadJS DataManager 초기화로 그리드 비어있음
- 재시도(같은 search 2회)는 캐시된 조건 + 이미 워밍된 DataManager로 즉시 응답

## 해결
**모든 PBM140MW 조회에 재시도 패턴 기본화:**
1. 1차 search 클릭 + 8초 대기
2. 행수 0이면 자동 재시도(같은 search 다시 클릭) + 8초 대기
3. 행수 확인

## 검증
- Edge 첫 신규 탭: 1차 0건 → 재시도 후 2건 정상
- Chrome: 1회 조회에 바로 정상 (캐시 워밍 상태)

## 영향
- 사용자 개입 없이 안정적 결과 보장
- 총 소요시간 8초 → 최대 16초 (2회 조회 시)
- 안정성 ↑
