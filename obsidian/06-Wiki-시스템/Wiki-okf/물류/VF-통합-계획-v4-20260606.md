# VF 바코드 시스템 ↔ VF 프로젝트 통합 계획 v4

> **기반:** v3 계획 + 검증팀 지적사항 5건 반영
> **반영 항목:** ①마이그레이션 전략 ②Phase 순서 ③UNIQUE 제약 ④취소/정정 ⑤에러처리/감사

---

## 1. 핵심 목적 (변경 없음)

**VF 바코드 = 입력 데이터를 바코드로 생성하는 메인 인터페이스**

- 생성 바코드 2종: (1) 입력 데이터 기반 바코드 (2) 송장 번호 바코드
- VF 바코드 입력 데이터 = 그날의 최종 출고 수량으로 VF 프로젝트에 반영

## 2. 데이터 모델 (③ UNIQUE 제약 수정 + ④ 취소/정정 로직 추가)

```sql
-- 제품 마스터 테이블
CREATE TABLE products (
    product_code TEXT PRIMARY KEY,
    product_name TEXT NOT NULL,
    category TEXT,
    is_active BOOLEAN DEFAULT TRUE,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 통합 출고 데이터 테이블
CREATE TABLE delivery_items (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    
    -- ③ 멱등성 키 (UNIQUE는 이것만 — 복합 UNIQUE 제거)
    idempotency_key TEXT UNIQUE NOT NULL,
        -- 생성 방식: SHA256(tracking_no || delivery_date || source)[:32]
        -- tracking_no=NULL인 경우 UUID 또는 auto-increment 사용
    
    -- 비즈니스 데이터
    product_code TEXT NOT NULL REFERENCES products(product_code),
    quantity INTEGER NOT NULL CHECK (quantity > 0),
    delivery_date DATE NOT NULL,
    time_slot TEXT,
    barcode_data TEXT,
    tracking_no TEXT,
    
    -- 출처
    source TEXT NOT NULL DEFAULT 'barcode' 
        CHECK (source IN ('vf_project', 'barcode', 'correction')),
    
    -- ④ 취소/정정
    is_cancelled BOOLEAN DEFAULT FALSE,
    cancelled_at DATETIME,
    cancel_reason TEXT,
    original_id INTEGER REFERENCES delivery_items(id),  -- 정정 시 원본 참조
    
    -- 입력자
    input_user_id TEXT,
    
    -- 타임스탬프
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    
    -- 인덱스
    INDEX idx_delivery_date (delivery_date),
    INDEX idx_product_code (product_code),
    INDEX idx_tracking_no (tracking_no),
    INDEX idx_source (source)
);

-- ⑤ 감사 로그 (변경 이력)
CREATE TABLE audit_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    table_name TEXT NOT NULL,
    record_id INTEGER NOT NULL,
    action TEXT NOT NULL CHECK (action IN ('INSERT', 'CANCEL', 'UPDATE', 'DELETE')),
    before_data TEXT,       -- JSON
    after_data TEXT,        -- JSON
    user_id TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_audit_record (table_name, record_id),
    INDEX idx_audit_date (created_at)
);

-- 통합 조회 뷰
CREATE VIEW vw_daily_delivery_total AS
SELECT 
    delivery_date,
    product_code,
    p.product_name,
    p.category,
    SUM(quantity) as total_quantity,
    COUNT(*) as record_count
FROM delivery_items d
JOIN products p ON d.product_code = p.product_code
WHERE d.is_cancelled = FALSE
GROUP BY delivery_date, product_code;
```

### ④ 취소/정정 상세 로직

| 시나리오 | 처리 |
|:---------|:------|
| **잘못 입력한 데이터 취소** | `POST /api/delivery/cancel` → is_cancelled=TRUE + audit_log |
| **취소 후 동일 데이터 재입력** | 새 idempotency_key 발급 (기존과 다른 키) → 정상 INSERT |
| **수량 정정** | 기존 레코드 CANCEL + 새 레코드 INSERT(source='correction', original_id=기존ID) |
| **취소된 레코드 원복** | is_cancelled=FALSE 재설정 (관리자 전용) |

## 4. ⑤ 에러 처리 및 동시성

| 시나리오 | 대응 |
|:---------|:------|
| 동일 idempotency_key 중복 요청 | UNIQUE 제약 → HTTP 409 + 기존 레코드 ID 반환 |
| 네트워크 장애 (프론트→백엔드) | 프론트 재시도 3회 (지수 백오프) |
| 동시성 (같은 키 동시 INSERT) | DB UNIQUE가 최종 방어선. 두 번째 요청=409 |
| 마이그레이션 중 장애 | 각 단계별 롤백 스크립트 실행 |
| Google Sheets 동기화 실패 | 로그 기록 + 다음 배치 재시도 (최대 3회) |

## 5. 마이그레이션 전략 상세 (① 구체화)

### idempotency_key 백필 방식

```python
import hashlib, uuid

def generate_key(tracking_no, delivery_date, source):
    if tracking_no:
        raw = f"{tracking_no}|{delivery_date}|{source}"
    else:
        raw = f"{uuid.uuid4()}|{delivery_date}|{source}"
    return hashlib.sha256(raw.encode()).hexdigest()[:32]

# 중복 발생 시: 타임스탬프 추가로 충돌 회피
#   raw = f"{tracking_no}|{delivery_date}|{source}|{timestamp}"
```
