# PBM110MW 납품요청: 수정 vs 신규등록 구분 (필수 규칙)

## 사건 (2026-06-08)
사용자가 "6월 10일로 해줘" (수정 요청) → 에이전트가 기존 데이터 조회 없이 바로 신규 INSERT + DELETE 시도 → DELETE가 실제로는 INSERT로 동작하여 중복 4건 발생 → 사용자가 직접 삭제 처리

## 원인
1. **"수정"과 "신규생성" 개념 혼동** — 사용자가 "일자 변경"이라고 했는데도 기존 조회 없이 새로 INSERT함
2. **사전 조회 누락** — 기존 등록 데이터(UNLOAD=20260609, 200개)가 있는지 먼저 확인하지 않음
3. **잘못된 DELETE 시도** — MOD='D'가 flag:true를 반환하지만 실제로는 INSERT가 실행됨 (MNG_GRD=05 차단)

## 교정된 규칙 (절대 준수)

### 수정 vs 신규등록 판단 기준

| 사용자 말 | 의미 | 처리 방법 |
|:---------|:-----|:---------|
| "신청해 줘", "등록해 줘", "요청해 줘" | **신규등록** | 조회 후 동일 데이터 없으면 INSERT (MOD='I') |
| "바꿔 줘", "수정해", "변경해", "6/10으로 해줘", "일자 변경" | **수정** | 조회 → 기존 행 찾기 → setValue로 값만 변경 → fn_save() |
| "삭제해", "지워" | **삭제** | 일반사용자는 DELETE 불가. 사용자에게 보고 |

### 필수 실행 순서 (PBM110MW 작업 전)

```
1. WPPS 로그인 확인
2. PBM110MW 페이지 접속 (browser_navigate)
3. SpreadJS 검색 실행 (조회 버튼 클릭)
4. 기존 데이터 확인 (REQ_DAT + UNLOAD_REQ_DAT 조합)
5. 동일 데이터 있음 → 수정 모드 / 없음 → 신규등록 모드
6. 진행 후 DB 재조회로 검증
```

### 절대 금지
- **❌ 수정 요청에 신규 INSERT (MOD='I') 하지 말 것**
- **❌ MOD='D'로 DELETE 시도하지 말 것** — flag:true지만 중복 INSERT 됨
- **❌ DELETE+INSERT로 수정을 대체하지 말 것**
- **❌ 사전 조회 없이 POST fetch() 하지 말 것**
