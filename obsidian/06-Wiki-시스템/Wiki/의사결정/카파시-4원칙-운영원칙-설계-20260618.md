## 왜 별도 스킬로 분리하는가

### 현재 문제: 운영원칙 분산

| 위치 | 포함된 카파시 4원칙 요소 |
|------|----------------------|
| `SOUL.md` §필수 규칙 | 절대 금지 (Surgical), 존댓말, 직접 검증 후 보고 |
| `mandatory-verification` SKILL.md | Skill-First Lock (Simple), Step 5 (Goal-oriented), Pitfall 40+개 |
| `rules.json` | §[DECISION LOCK], §[PRE-FLIGHT], ASK rule (2026-06-11 일부 제거됨) |
| `meta-harness` | 라우팅 우선순위 (Simple) |

**문제 4가지:**
1. "Think하라"는 원칙이 어디 정의되어 있는지 찾기 어려움
2. 새 Pitfall이 생길 때 어디에 추가할지 매번 판단
3. mandatory-verification Pitfall 40+개에 카파시 flavor 4개 추가 시 가독성 ↓
4. 다른 스킬에서 참조 시 mandatory-verification 전체 로드 (9000+ chars)

### 해결: 단일 스킬로 통합

```
📁 dogfood/karpathy-4-principles/
   └── SKILL.md          ← 4원칙 3 flavor의 단일 진실 공급원 (SSOT)
```

**장점:**
- 4원칙만 따로 보면 됨 (가독성)
- 다른 스킬에서 `skill_view('karpathy-4-principles')`로 가벼움 (1~2KB)
- 수정 시 1000자 파일 패치로 끝
- "이건 4원칙, 이건 검증 절차" 명확한 경계

## 3 Flavor 설계

### Flavor D8 — "Think" (사고 강제)

**원칙:** 작업 전 목표·제약·입출력을 명시한 뒤 행동

| | 내용 |
|---|------|
| **무엇을 한다** | 목표·제약·입출력 먼저 명시 |
| **위반 예** | "추측으로 시작" / "일단 해보고 생각" |
| **준수 예** | "이 작업의 목표 = X, 제약 = Y, 입출력 = Z" |
| **관련 기존 규칙** | 절대 금지 #7 (추측/가정 금지), Holographic [0.5] "확인하고 말해" |

### Flavor D7 — "Simple" (단순성 강제)

**원칙:** 가장 간단한 해결책 우선

| | 내용 |
|---|------|
| **무엇을 한다** | 기존 도구/스킬 확인 후 그대로 사용. 새로 만들기 전 검증. |
| **위반 예** | "기존 스킬 무시하고 새로 만들기" / "더 좋은 방법이라 새로 작성" |
| **준수 예** | "skills_list() 확인 → 50%+ 커버 시 기존 사용" |
| **관련 기존 규칙** | Skill-First Lock (mandatory-verification), Holographic "네 마음대로 하지 마" |

### Flavor INV — "Surgical" (정밀성 강제)

**원칙:** 변경은 필요한 부분만. 의도치 않은 변경/소실 금지

| | 내용 |
|---|------|
| **무엇을 한다** | 변경 전 상태 확인 → 필요한 부분만 수정 → 검증 |
| **위반 예** | "git checkout으로 사용자 변경사항 소실" / "초기화해/리셋해 자동 실행" / "원본 그대로 지시 + 임의 수정" |
| **준수 예** | "git status 먼저 → stash 백업 → 작업" |
| **관련 기존 규칙** | git-safety-windows, 절대 금지 #13 (원본 그대로 추가 수정 금지), 절대 금지 #11 (보고 전 직접 확인) |

### Flavor NOTICE — "Goal-oriented" (목표지향 강제)

**원칙:** 모든 작업은 3중 완료 (Wiki + README + Git push)

| | 내용 |
|---|------|
| **무엇을 한다** | 작업 완료 = Wiki 기록 + README 갱신 + Git push. 부분 완료 보고 금지. |
| **위반 예** | "됐다고 보고하고 검증 생략" / "Wiki만 갱신하고 push 누락" |
| **준수 예** | "API GET 조회 → 실제 값 인용 → Wiki 기록 → README 갱신 → Git push" |
| **관련 기존 규칙** | SOUL.md §6 3중 완료 의무 |

## 결정고정(§7)과 충돌 방지

### 충돌 지점
- D8(Think) = "사고를 위해 한 번 더 묻기" 위험
- 결정고정 = "사용자 '네' = 즉시 실행, 재질문 금지"

### 해결 — 명문화
- D8(Think)은 **스스로 사고**이지 "사용자에게 다시 묻는 것"이 아님
- 결정고정(§7)이 **최상위 우선순위**임을 SKILL.md 첫 줄에 명시

### SKILL.md 첫 줄 (안)
```markdown
# 카파시 4원칙 운영원칙 3 Flavor

> **이 스킬은 결정고정(§7) 아래에 있다.**
> 사용자가 "네/진행해/OK" 등으로 1회 이상 확인한 경우 즉시 실행한다.
> D8(Think)은 사고 강제이지 사용자 재질문이 아니다.

## Flavor D8 — "Think" (사고 강제)
...
```

### rules.json
```json
{"role":"system","content":"§[KARPATHY-4P] D8(Think)·D7(Simple)·INV(Surgical)·NOTICE(Goal-oriented). 결정고정(§7) 최우선."}
```

```markdown
## 6. 카파시 4원칙 운영원칙

- D8(Think): 사고 강제
- D7(Simple): 단순성 강제
- INV(Surgical): 정밀성 강제
- NOTICE(Goal-oriented): 3중 완료 의무
```

## Phase 1 작업 순서 (사용자 승인 후)

1. `dogfood/karpathy-4-principles/SKILL.md` 작성 (~1KB)
2. `mandatory-verification/SKILL.md`에 "§0.5 karpathy-4-principles" 링크 추가
3. `SOUL.md` §6 신설 (카파시 4원칙)
4. `~/.hermes/rules.json`에 `§[KARPATHY-4P]` 항목 추가
5. Gateway 재시작 (`hermes gateway restart` 외부 터미널)
6. Wiki 본 문서 갱신 (적용 완료 기록)
7. Git push
