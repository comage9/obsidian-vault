## 2. 외부 표준 — "Harness Engineering" 이론 (2026 SOTA)

### 2-1. arXiv "Harness Engineering as Categorical Architecture"

**핵심 인용:**
> "The four pillars of agent externalization: **Memory, Skills, Protocols, Harness Engineering** map onto the triple's components: **Memory** as coalgebraic state, **Skills** as operad-composed objects, **Protocols** as syntactic wiring G, and the full **Harness** as the Architecture itself."

→ **Memory = Harness의 4대 기둥 중 하나이며, 별도 외재화 시스템**

### 2-2. arXiv "Externalization in LLM Agents" (Unified Review)

**핵심 인용:**
> "memory externalizes state across time, skills externalize procedural expertise, protocols externalize interaction structure, and **harness engineering serves as the unification layer that coordinates them into governed execution**."

→ 표준 정의:
- **Memory = 시간에 따른 상태 외재화**
- **Skills = 절차적 전문성 외재화**
- **Protocols = 상호작용 구조 외재화**
- **Harness = 이 3가지를 governed execution으로 통합하는 계층**

### 2-3. 표준 Auto-Harness 시스템 (메모리 보유)

arXiv 검색에서 **6개 이상의 표준 auto-harness 시스템** 발견:

| 시스템 | 장기기억 보유? | 핵심 |
|--------|--------------|------|
| **Meta-Harness** (곰너이 님 시스템과 동일 이름) | ✅ yes (메모리 자동 갱신) | "auto-harness systems such as A-Evolve, GEPA, and **Meta-Harness** improve LLM agents by optimizing prompts, skills, **tools, memories**, and supporting infrastructure" |
| **A-Evolve** | ✅ yes | A-EVO-Lab의 진화형 하네스 |
| **GEPA** | ✅ yes | Genetic-Pareto 진화 |
| **Adaptive Auto-Harness** | ✅ yes | task-wise adaptation, harness tree, solve-time routing |
| **HarnessFix** | ✅ yes | trace-guided, HTIR (Harness-aware Trace IR) |
| **Harness-1** (검색 에이전트 20B) | ✅ yes | state-externalizing harness, candidate pool, importance-tagged curated set, evidence links, verification records |
| **HarnessForge** | ✅ yes | harness-policy co-evolution |
| **Life-Harness** | ✅ yes | lifecycle-aware runtime, 116/126 settings 개선 |
| **Harnessing Generalist Agents (TimeClaw)** | ✅ yes | "episodic multimodal memory for retrieving relevant reasoning traces" |

### 2-4. 핵심 발견: 곰너이 님의 `meta-harness`는 "Memory를 가진 Meta-Harness"가 아님

arXiv 명시:
> "Meta-Harness ... improve LLM agents by optimizing prompts, skills, tools, **memories**, and supporting infrastructure"

표준 Meta-Harness는 **memories를 자동으로 최적화**합니다. 곰너이 님 시스템의 `meta-harness` 스킬은 이름만 같고, **실제로는 라우팅만 함**. 표준 Meta-Harness의 핵심 기능인 memory optimization이 빠져 있습니다.

## 3. 표준 하네스의 Memory 보관 방법 (실측 사례)

### 3-1. Harness-1 (20B 검색 에이전트, RL 학습)

> "The harness maintains environment-side **working memory**, including:
> - candidate pool
> - importance-tagged curated set
> - compact evidence links
> - verification records
> - compressed and deduplicated observations
> - **budget-aware context rendering**"

→ **6개 메모리 구성요소**: 풀, 큐레이션 셋, 링크, 검증 기록, 압축 관찰, 예산 인지 렌더링

### 3-2. TimeClaw (시계열 + 멀티모달)

> "executable temporal tools ... experience-driven capability evolution for creating reusable analytical routines, and **episodic multimodal memory** for retrieving relevant reasoning traces"

→ **3-tier 메모리**: 도구(절차), 진화 스킬(절차 기억), episodic memory(에피소드 기억)

### 3-3. HarnessFix (HTIR, Trace-guided)

> "Harness-aware Trace Intermediate Representation (HTIR), which normalizes fragmented trajectory evidence and captures step-level provenance and control-flow relations"

→ **Trajectory-based**: 실행 트레이스 → HTIR로 정규화 → 단계별 provenance 저장

### 3-4. Memory Bank (HarnessFix의 보완)

> "consolidates recurring diagnoses into actionable **flaw records**"

→ **결함 레코드** = 경험을 통한 자동 학습된 함정 목록 (곰너이 님의 PITFALL과 같은 개념!)

## 4. 곰너이 님 시스템 vs 표준 Auto-Harness 메모리 매핑

| 표준 Auto-Harness 메모리 | 곰너이 님 시스템 매핑 | 일치도 |
|------------------------|---------------------|--------|
| **Working memory** (현재 컨텍스트) | 컨텍스트 윈도우 (단기) | ✅ |
| **Episodic memory** (시간 인덱싱 경험) | session_search (DB) | ⚠️ 부분 (binary, markdown 아님) |
| **Semantic memory** (의미/사실) | MEMORY.md, SOUL.md, USER.md | ✅ |
| **Procedural memory** (절차/스킬) | SKILL.md (89개) | ✅ |
| **Candidate pool** (검색 후보) | 없음 | ❌ |
| **Evidence links** (인용 링크) | Wiki 내부 [[링크]] | ⚠️ 부분 |
| **Verification records** (검증 이력) | git commit history | ✅ |
| **Compressed observations** (요약) | 없음 | ❌ |
| **Flaw records** (자동 학습 함정) | PITFALL 섹션 (수동) | ⚠️ 부분 |
| **Importance-tagged** (중요도) | 없음 | ❌ |
| **Budget-aware context** (예산 인지) | 없음 | ❌ |
| **Self-evolving** (자동 진화) | 자가학습 cron 5개 (수동 점검) | ⚠️ 부분 |

### 핵심 격차:
- 곰너이 님 시스템은 3-tier (Episodic/Semantic/Procedural) 메모리는 갖고 있으나
- **6개 이상의 표준 구성요소** 중 3-4개만 구현
- **중요도, 예산 인지, 후보 풀, 압축** = 미구현

> 💡 **실용적 결론:** 위 arXiv 논문들은 외부 연구 참고용입니다. 곰너이 님 시스템의 장기기억 개선은 **곰너이 님 시스템의 harness 스킬(harness-cross-verify + meta-harness) 자체에는 장기기억 기능이 없습니다.** 이미 곰너이 님은 Hermes Built-in Memory + Wiki + Session Search로 표준 3-tier를 구현 중이며, 외부 Provider(Holographic 등) 도입 시 추가 개선 가능합니다.

## 5. 곰너이 님 시스템의 강점 (표준 대비)

표준 Auto-Harness 대비 곰너이 님 시스템이 **우월한 부분**:

1. **Obsidian + Git = 영속성 + 버전 관리 + 크로스 디바이스** — 표준은 vector DB 의존, 곰너이 님은 자체 호스팅

3. **절대 금지/각서 박제** — 실패 패턴 영구 보존
4. **89개 명시적 스킬** — 표준 시스템은 1-2개 통합 스킬
5. **카르파티 LLM Wiki 패턴** — 표준 harness의 memory를 인간이 읽을 수 있는 markdown으로 보존
6. **자가학습 cron 5개** — 표준 harness의 self-evolution을 시간 기반으로 자동화
