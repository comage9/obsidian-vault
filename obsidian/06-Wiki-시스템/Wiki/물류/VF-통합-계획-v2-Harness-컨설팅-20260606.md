## [Model 1] M3 (MiniMax) — 메인 컨설팅 분석
⏱ 60.6s

### 1. 데이터 통합 방식 평가

**결론:** 방향은 옳지만, "추가"의 의미를 더 엄격하게 정의할 필요가 있음

**✅ 적절한 측면:**
- 데이터 무손실: 기존 VF 프로젝트 데이터를 훼손하지 않는 안전한 접근
- 롤백 가능성: 신규 데이터 별도 분리로 문제 시 즉시 분리 가능
- 점진적 전환: 단계적 통합 가능
- 비즈니스 연속성: 업무 중단 없이 통합 가능

**⚠️ 보완 필요 사항:**
1. "추가"는 이벤트 소싱(Event Sourcing)에 가까움 — 모든 변경을 불변 이벤트로 기록
2. 충돌 시나리오 정의 필요:
   - 같은 송장번호 중복 입력 → 멱등성 보장
   - 같은 시간대 다른 수량 → 비즈니스 룰 정의
   - 기존 데이터 수정 필요 시 → 정합성 정책
   - 신규 데이터 취소/반품 → 취소 이벤트 설계
3. **권장: 멱등성 키(idempotency_key) 도입**, source를 ENUM으로 명확화

### 2. DB 스키마 설계 평가

**✅ 잘 설계된 부분:**
- 기존 테이블 보존 + 신규 테이블 추가 = 데이터 격리 측면 안전
- 통합 뷰 패턴 = CQRS 패턴 Read Model과 유사

**⚠️ 보완 필요:**
1. **멱등성 보장 컬럼(idempotency_key UNIQUE) 필수** — 바코드 중복 스캔 방지
2. **제품 정보 별도 테이블 정규화** 필요 (현재 중복 구조)
3. **소프트 삭제(is_cancelled)** 및 **인덱스** 설계 필요
4. **입출고 이력 추적**을 위한 메타데이터(input_user_id, input_device)

## [Model 2] DeepSeek-V4-flash — 비판적 검토
⏱ 23.9s

### 1. 이중 테이블 방식 평가

**단점 및 우려사항:**
- **중복 데이터 위험:** 동일 (date, product, time_slot)이 두 테이블에 존재 시 UNION ALL에서 이중 집계
- **제품 메타데이터 동기화 문제:** product_code/name/category가 두 테이블에 중복 저장. VF 프로젝트에서 제품 정보 변경 시 정합성 깨짐
- **조회 복잡성:** 출처별 분리 조회 시 복잡한 쿼리 필요

**대안 제안:**
> **단일 테이블 + source 컬럼** 방식 권장. 기존 테이블에 `source TEXT` 컬럼 추가.
> 유니크 키를 `(delivery_date, product_code, time_slot)`으로 설정하여 중복 원천 차단.

### 2. UNION ALL 뷰 무결성 문제

- 동일 조합 양쪽 존재 시 **실제 출고량보다 많은 수량** 계산 → 비즈니스 오류
- NULL 처리 필요: `COALESCE(quantity, 0)`
- **FK 제약조건 또는 입력 시 제품 유효성 검증** 필수

### 3. Google Sheets 동기화

- **양방향 동기화 충돌 위험** — Sheets 수동 수정 + DB 변경 충돌 시 데이터 소실
- **권장: 단방향(DB → Sheets)**, DB를 단일 진실 공급원(Source of Truth)으로
- 정기 배치(5분마다) 동기화, 실패 시 재시도 로직 필요

## 최종 권장 사항 (종합)

### 1. DB 설계 변경
- **단일 테이블 + source 컬럼** 우선 고려 (DS 제안)
- 또는 **2개 테이블 유지 시** 멱등성 키 + FK 제약조건 필수 추가
- **멱등성 키(idempotency_key UNIQUE)** 반드시 도입
- **제품 정보 정규화** (별도 products 테이블)

### 2. Google Sheets 전략
- **단방향 동기화(DB → Sheets)** 로 변경
- Sheets는 순수 백업 + 외부 보고용으로만 사용

### 3. Phase 우선순위 조정
- Phase A (React 전환) → 기존대로 1순위
- Phase B (데이터 통합) → **멱등성·정합성**을 먼저 설계 후 구현
- Phase C (고도화) → 선택 유지

### 4. 추가 필요한 설계
- 취소/정정 이벤트 처리 (소프트 삭제 또는 보정 이벤트)
- 중복 입력 방지 로직 (프론트엔드 + DB 이중 체크)
- 동기화 실패 시 복구 절차
