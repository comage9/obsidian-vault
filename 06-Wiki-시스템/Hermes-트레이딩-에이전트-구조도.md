## 1. 현황 정리

### 현재 ki-ai-trader 시스템
```
Hermes (사용자 명령) → ki_project_monitor.py (Python) → kiwoom_api.py (키움 OpenAPI)
```

**현재 문제점:**
- 스크리닝 → 신호 → 주문 실행이 Python으로 하드코딩됨
- Hermes는 단순 알림 수신자 역할 (가격 조회만 가능)
- 주문 실행, 포트폴리오 조정, 리스크 판단 → 전부 Python이 처리
- 다중 에이전트 협업 불가 (Bull/Bear/Trader/Risk 분석을 동시에 불가)
- 새벽/공휴일 등에는 Python 스크립트가 필요한 시기에 실행 불가

### 목표
```
Hermes (오케스트레이터) → 다중 AI 에이전트 협업 → 키움 API 직접 주문 실행
```

## 2. Hermes 트레이딩 에이전트 — 3단계 로드맵

### 🏗️ 단계 1: Hermes ↔ Kiwoom API 연동 (기초 계층)

**목표:** Hermes가 키움 계좌를 직접 읽고, 사용자가 승인하면 주문을 내는 구조

```
┌─────────────────────────────────────────────────────┐
│                    Hermes Agent                       │
│  ┌─────────────┐   ┌──────────────┐   ┌──────────┐  │
│  │ 키움 도구   │   │ 분석 도구    │   │ 주문 도구│  │
│  │ (새로 제작) │   │ (기존 재사용) │   │ (새로 제작)│  │
│  └──────┬──────┘   └──────┬───────┘   └────┬─────┘  │
└─────────┼──────────────────┼────────────────┼────────┘
          │                  │                │
          ▼                  ▼                ▼
    get_balance()      analyze()         place_order()
    get_holdings()     screen()          cancel_order()
    get_price()        news_search()      get_order_list()
```

**단계 1 구현:**

#### (1) Kiwoom MCP 서버 구축
```python
# kiwoom_mcp_server.py
# Hermes가 kiwoom_api.py를 직접 호출하는 MCP 서버
# 키움 OpenAPI Plus (HTTP REST) 사용

Tools:
- kiwoom_get_balance() → 예수금/총자산
- kiwoom_get_holdings() → 보유 종목/수량/손익
- kiwoom_get_price(code) → 현재가/변동률
- kiwoom_place_buy(code, qty, price) → 매수 주문
- kiwoom_place_sell(code, qty, price) → 매도 주문
- kiwoom_get_order_list() → 미체결 주문
- kiwoom_cancel_order(order_id) → 주문 취소
```

#### (2) Kiwoom Skill 제작
```markdown
# kiwoom-mcp.md
키움 OpenAPI Plus 직접 제어 스킬
- MCP 서버 endpoint, 인증 방법
- 각 tool의 사용법과 예시
- 안전 장치: 주문 전 사용자 승인 필수
```

#### (3) 사용자 승인 워크플로우
```
Hermes: "NVDA 매수 제안: 10주 @ 900달러"
         사용자가 Telegram/대화에서 "승인" → Hermes가 place_buy() 호출
```

**예상 소요 시간:** 2~3시간 (키움 API 인증 + MCP 서버 + 스킬)

### 🤖 단계 2: 다중 에이전트 협업 (Hermes Swarms 통합)

**목표:** 7개 에이전트가 실시간으로 분석 → Hermes가Orchestra

```
사용자: "NVDA 검토해줘"

Hermes 오케스트레이터
    │
    ├─ [Analyst 1: Fundamenta] ──→ 재무 분석 ──→ 보고서
    ├─ [Analyst 2: Sentiment]  ──→ 센티먼트 ──→ 보고서
    ├─ [Analyst 3: News/Macro] ──→ 뉴스/거시 ──→ 보고서
    ├─ [Analyst 4: Technical]  ──→ 기술적 분석 ──→ 보고서
    │
    ▼ (동시에 병렬 실행, 서로 모름)
    
Bull Researcher ←── 4개 보고서
Bear Researcher ←── 4개 보고서
    │
    │ (서로 3라운드 논쟁)
    ▼
Trader ←── Bull/Bear 논쟁 결과
Risk Manager ←── Trader 결정 검토
    │
    ▼
Portfolio Manager ←── 최종 결정
    │
    ▼
Hermes → 사용자에게 보고서 전달 → "NVDA BUY 제안, 10주 @ 900달러"
    │
    ▼ (사용자 승인)
키움 API: 매수 주문 실행
```

**Swarms 스킬 (`hermes-swarm-hedgefund`) 재사용:**
- Cerebras 무료 모델 (4개 분석가 + Bull/Bear/Trader/Risk/PM)
- 기존 스킬 그대로 사용 가능
- **새로 추가:** kiwoom tool call을 각 에이전트 프롬프트에 통합

**예상 소요 시간:** 1~2시간 (프롬프트 조정 + tool call 연결)

## 3-B. 백테스트 결과 (2026-05-04)

**데이터:** Yahoo Finance, 10개 종목, 2026-04-06 ~ 2026-05-04 (약 19일)
**방법:** RSI, Bollinger, CCI, Stoch, VolumeRatio, PriceChange 기반 점수 매수 + 손절/트레일링/익절

### 파라미터 검증 결론

| 항목 | 기존 | **2026-05-04 검증** | **2026-05-05 변경** |
|------|------|---------------------|----------------------|
| 손절 | -3% | -7% | -7% (유지) |
| 익절 | +5% | +10% (고정) | **ATR 기반 (2×ATR)** |
| 트레일링 | 5% | 7% | 7% (유지) |
| RSI 기준 | 35 | 30 이하 | 30 이하 (유지) |

**상위 조합 (수익률 기준):**
- 손절 -7% + 트레일링 7% + 익절 10%: **+7.85%, 85.7% 승률**
- 손절 -7% + 트레일링 10% + 익절 10%: **+7.85%, 85.7% 승률**

**핵심 발견:**
- 익절 10%가 손절 범위(-3%~10%)보다 수익률에 더 큰 영향
- 트레일링 3%는 모두 좋지 않은 결과 (5%+ 필요)
- -3% 손절은 너무 잦은 매도 (21회/47% 승률) → -7% 손절이 효율적

### ATR 기반 익절 (2026-05-05 적용)

**기존:** 전 종목 고정 +10% 익절 → 변동성 큰 종목에서 상승분을 놓침
**신규:** `익절가 = 매수가 + (ATR_14 × 2)`

| 항목 | 값 |
|------|-----|
| ATR 기준 | 14일 ATR (yfinance) |
| 익절 배수 | ATR × 2 |
| 계산 시점 | 매수 시 (PositionTracker 생성 시) |
| 슬리피지 | 0.15% |

**적용 예시:**

| 종목 | 매수가 | ATR(14) | 익절가 | 익절 수익률 |
|------|--------|---------|--------|------------|
| 삼성전자 | 70,000 | 800 | 71,600 | +2.3% |
| HMM | 20,000 | 1,425 | 22,850 | +14.3% |
| Hyundai | 180,000 | 3,000 | 186,000 | +3.3% |

**동작 흐름:**
```
상승 중 → 최고가 갱신 → 관망 (HOLD)
현재가 ≥ 익절가 → 🎯 ATR 익절! → 전량 매도
현재가 < 익절가, drawdown ≥ 7% → ⚠️ 트레일링 → 분할/전량 매도
매수가 -7% 이하 → 🔴 손절 → 전량 매도
```

## 4. 4대 버그/개선 수정 (2026-05-05) — 멀티 에이전트 감사

### 감사 에이전트 3개 병렬 실행
| 에이전트 | 역할 | 발견 |
|----------|------|------|
| Data Collector | 백테스트 엔진 구축 | `backtest_engine.py` + 데이터 수집 파이프라인 |
| Code Auditor | trailing_stop_trader.py 감사 | 4가지 개선점 |
| ATR Researcher | ATR 배수 검증 | ×2 유지 권장 (+0.71%, MDD 4.92%) |

### 발견된 문제 및 수정

| # | 심각도 | 유형 |发现问题 | 수정 내용 |
|---|--------|------|---------|----------|
| 1 | ⚠️ **치명** | 로직 버그 | `atr_target`이 `buy_price` 기준 고정 → 최고가 갱신 시 재계산 안 됨 | `update_highest()`에서 `atr_target = highest_price + atr×2` 동적 갱신 |
| 2 | ⚠️ **치명** | 슬리피지 | `SLIPPAGE=0.0015` 설정済だが `execute_sell()` 미적용 | `sell_price = price × (1 - SLIPPAGE)` 적용 |
| 3 | ⚠️ 중요 | 우선순위 | 손절이 마지막 체크 → 급락 시 지연 | ①손절 → ②ATR익절 → ③트레일링 순서明确规定 |
| 4 | ⚠️ 중요 | 재시도 부재 | API 주문 1회 실패 시 복구 불가 | 최대 3회 재시도 (지수 백오프 5s/10s/15s) + 토큰 만료 재인증 |

**수정 후 동작 흐름:**
```
① 손절 체크 → -7% 이하 → 즉시 전량 매도 (최우선)
② ATR 익절 → 현재가 ≥ (최고가 + ATR×2) → 분할 또는 전량
③ 트레일링 → 현재가 < 최고가 -7% → 분할 또는 전량
④ 관망 → 상승 추세 유지
```
